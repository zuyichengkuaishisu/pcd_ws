# 山猫 M20 机器人本体监控协议（正式交付版）

> 适用对象：机器人本体与上位机、外部板卡或外部系统的协议联调、接口开发与交付说明。\
> 适用型号：`M20` / `M20 Pro`（巡检与导航相关接口仅 `M20 Pro` 支持）。\
> 数据格式：支持 `JSON` 与 `XML`，推荐优先使用 `JSON`。\
> 传输协议：支持 `TCP` 与 `UDP`。

***

## 1. 文档说明

本文档对机器人本体监控协议进行了重新整理，目标是将原始协议说明重构为适合交付、对接和二次开发的标准文档。文档重点包括：

- 通信基础约定
- APDU / ASDU 报文结构
- 控制类接口
- 状态类接口
- 巡检 / 导航类接口
- 通用调用状态响应
- 常见错误码与导航异常码

如无特殊说明：

- `Type` 表示消息类型
- `Command` 表示消息命令码
- `Time` 使用本地时区时间，格式为 `YYYY-MM-DD HH:MM:SS`
- `Items` 为业务参数载荷
- 示例优先给出 `JSON`；`XML` 与其字段同名、结构一一对应

## 2. 通信基础

### 2.1 通信角色

协议使用过程中，双方角色如下：

- 机器人本体：`TCP/UDP 服务端`
- 上位机、外部板卡或外部系统：`TCP/UDP 客户端`

### 2.2 默认端口

| 协议  | 服务端地址          |      端口 |
| --- | -------------- | ------: |
| UDP | `10.21.31.103` | `30000` |
| TCP | `10.21.31.103` | `30001` |

### 2.3 协议层次

本协议位于 OSI 七层模型中的应用层，对应关系如下：

| 层级         | 内容              |
| ---------- | --------------- |
| 应用层（第 7 层） | 本协议             |
| 传输层（第 4 层） | TCP/IP 或 UDP/IP |
| 网络层（第 3 层） | IP              |
| 链路层（第 2 层） | 以太网             |
| 物理层（第 1 层） | 物理介质            |

备注：第 5 层和第 6 层未单独使用。

### 2.4 交互机制

本协议以“请求 / 响应”为主，部分状态信息会在满足条件后主动上报。

典型流程如下：

1. 上位机向机器人发送请求报文。
2. 机器人回送对应响应报文。
3. 若上位机持续发送心跳，机器人还会主动上报实时状态和异常状态。

## 3. 报文结构

### 3.1 APDU 结构

APDU（Application Protocol Data Unit，应用协议数据单元）采用以下结构：

```text
APDU = 协议头部 + ASDU
```

约束如下：

- 每个 APDU 最多携带 `1` 个 ASDU
- 协议头部长度固定为 `16` 字节

### 3.2 协议头部定义

| 序号 | 字段      | 长度（字节） | 说明                         | 备注               |
| -- | ------- | -----: | -------------------------- | ---------------- |
| 1  | 同步字符    |      1 | `0xeb`                     | 固定               |
| 2  | 同步字符    |      1 | `0x91`                     | 固定               |
| 3  | 同步字符    |      1 | `0xeb`                     | 固定               |
| 4  | 同步字符    |      1 | `0x90`                     | 固定               |
| 5  | 长度      |      2 | ASDU 字节长度                  | 小端字节序；最大 `65535` |
| 6  | 报文 ID   |      2 | 请求帧唯一标识                    | 小端字节序；响应帧沿用同一 ID |
| 7  | ASDU 格式 |      1 | `0x00` = XML，`0x01` = JSON | 建议使用 `0x01`      |
| 8  | 预留      |      7 | `0x00`                     | 固定填 0            |

### 3.3 ASDU 通用字段

| 字段        | 含义     |
| --------- | ------ |
| `Type`    | 消息类型   |
| `Command` | 消息命令码  |
| `Time`    | 报文发送时间 |
| `Items`   | 消息参数项  |

### 3.4 通用 JSON 示例

```json
{
  "PatrolDevice": {
    "Type": 1002,
    "Command": 1,
    "Time": "2023-01-01 00:00:00",
    "Items": {}
  }
}
```

### 3.5 通用 XML 示例

```xml
<?xml version="1.0" encoding="UTF-8"?>
<PatrolDevice>
  <Type>1002</Type>
  <Command>1</Command>
  <Time>2023-01-01 00:00:00</Time>
  <Items />
</PatrolDevice>
```

## 5. 控制类接口

### 5.1 心跳指令

**接口用途**

用于建立保活关系，并触发机器人向当前客户端主动上报实时状态和异常状态。

**接口定义**

| 项目      | 值     |
| ------- | ----- |
| Type    | `100` |
| Command | `100` |
| 消息名称    | 心跳指令  |
| 方向      | 请求    |

**请求参数**

无。

**建议**

- 建议发送频率不低于 `1Hz`
- 机器人会向持续发送心跳的 IP 和端口上报状态信息

**JSON 请求示例**

```json
{
  "PatrolDevice": {
    "Type": 100,
    "Command": 100,
    "Time": "2023-01-01 00:00:00",
    "Items": {}
  }
}
```

**接口定义**

| 项目      | 值      |
| ------- | ------ |
| Type    | `1101` |
| Command | `5`    |

| 参数     | 含义      | 类型  | 取值                         |

### 5.7 自主充电

**接口定义**

{

    "Type": 2,
    "Command": 24,
    "Time": "2023-01-01 00:00:00",
    "Items": {

    }
  }

## 6. 状态类接口

### 6.1 总体说明

状态类接口与心跳联动：机器人会向持续发送心跳指令的客户端主动上报状态。

### 6.2 基础状态上报

**接口定义**

| 项目      | 值           |
| ------- | ----------- |
| Type    | `1002`      |
| Command | `6`         |
| 消息名称    | 获取机器人基础状态信息 |
| 方向      | 主动上报        |
| 频率      | `2Hz`       |

**响应字段：`Items.BasicStatus`**

| 字段                 | 含义       | 类型     | 取值说明                                                     |
| ------------------ | -------- | ------ | -------------------------------------------------------- |
| `MotionState`      | 机器人运动状态  | int    | `0` 空闲；`1` 站立；`2` 关节阻尼/软急停；`3` 开机阻尼；`4` 趴下；`17` RL 控制    |
| `Gait`             | 机器人步态    | int    | `0x1001` 基础；`0x1002` 高台；`0x3002` 平地；`0x3003` 楼梯          |
| `Charge`           | 充电状态     | int    | `0` 空闲；`1` 前往充电桩；`2` 充电中；`3` 退出充电桩；`4` 机器人异常；`5` 在桩上但未充电 |
| `HES`              | 硬急停状态    | int    | `0` 未触发；`1` 已触发                                          |
| `ControlUsageMode` | 使用模式     | int    | `0` 常规；`1` 导航；`2` 辅助                                     |
| `Direction`        | 前进正方向    | int    | `0` 正向为前进方向；`1` 后向为前进方向                                  |
| `OOA`              | 辅助模式避障状态 | int    | `0` 未启动；`1` 空闲；`2` 未触发避障；`3` 主动避障中                       |
| `PowerManagement`  | 电源管理模式   | int    | `0` 常规模式；`1` 单电池模式                                       |
| `Sleep`            | 休眠状态     | int    | `0` 未休眠；`1` 已休眠；`2` 进入休眠中                                |
| `Version`          | 设备版本     | string | `STD` = 山猫 M20；`PRO` = 山猫 M20 Pro                        |

**补充说明**

- `Direction` 的“前进正方向”指下发 `X` 轴正值时机器人的移动方向
- 当 `Direction = 1` 时，机器人以后向为前进正方向；后向定义为带硬急停一侧

**JSON 响应示例**

```json
{
  "PatrolDevice": {
    "Type": 1002,
    "Command": 6,
    "Time": "2023-01-01 00:00:00",
    "Items": {
      "BasicStatus": {
        "MotionState": 0,
        "Gait": 4097,
        "Charge": 0,
        "HES": 0,
        "ControlUsageMode": 0,
        "Direction": 0,
        "OOA": 0,
        "PowerManagement": 0,
        "Sleep": 0,
        "Version": "STD"
      }
    }
  }
}
```

### 6.3 运控状态上报

**接口定义**

| 项目      | 值           |
| ------- | ----------- |
| Type    | `1002`      |
| Command | `4`         |
| 消息名称    | 获取机器人运控状态信息 |
| 方向      | 主动上报        |
| 频率      | `10Hz`      |

**响应结构**

- `Items.MotionStatus`：整机运动状态
- `Items.MotorStatus`：关节状态

**`MotionStatus`** **字段**

| 字段                       | 含义       | 类型    | 单位      |
| ------------------------ | -------- | ----- | ------- |
| `Roll` / `Pitch` / `Yaw` | 姿态角      | float | `rad`   |
| `OmegaZ`                 | Z 方向角速度  | float | `rad/s` |
| `LinearX` / `LinearY`    | 线速度      | float | `m/s`   |
| `Height`                 | 当前机身高度   | float | `m`     |
| `Payload`                | 无效参数     | -     | -       |
| `RemainMile`             | 预计剩余续航里程 | float | `km`    |

**`MotorStatus`** **字段**

| 字段       | 含义           | 类型           | 单位              |
| -------- | ------------ | ------------ | --------------- |
| `Joint`  | 长度为 16 的关节数组 | array<float> | `rad` / `rad/s` |
| `*HipX`  | 侧摆关节角度       | float        | `rad`           |
| `*HipY`  | 髋关节角度        | float        | `rad`           |
| `*Knee`  | 膝关节角度        | float        | `rad`           |
| `*Wheel` | 足轮转速         | float        | `rad/s`         |

**Joint 数组顺序**

1. `LeftFrontHipX`
2. `LeftFrontHipY`
3. `LeftFrontKnee`
4. `LeftFrontWheel`
5. `RightFrontHipX`
6. `RightFrontHipY`
7. `RightFrontKnee`
8. `RightFrontWheel`
9. `LeftBackHipX`
10. `LeftBackHipY`
11. `LeftBackKnee`
12. `LeftBackWheel`
13. `RightBackHipX`
14. `RightBackHipY`
15. `RightBackKnee`
16. `RightBackWheel`

**命名说明**

- `Left / Right`：左侧 / 右侧，其中左侧为电池仓所在一侧
- `Front / Back`：前侧 / 后侧，其中后侧为硬急停按钮所在一侧

### 7.1 初始化和重置定位

**接口定义**

| 项目      | 值        |
| ------- | -------- |
| Type    | `2101`   |
| Command | `1`      |
| 消息名称    | 初始化和重置定位 |

**请求参数**

| 参数                       | 含义             | 类型    | 单位    |
| ------------------------ | -------------- | ----- | ----- |
| `PosX` / `PosY` / `PosZ` | 地图坐标系下的位置      | float | `m`   |

**像素坐标换算公式**

```text
x_p = (PosX - X0) / Res
y_p = H - (PosY - Y0) / Res
```

说明：

- `(x_p, y_p)` 取整后为像素坐标

**响应参数**

| 参数          | 含义      | 类型  | 取值            |

**注意事项**

- 定位重置需要一定时间
- 即时返回 `ErrorCode = 1` 不一定代表实际重定位失败
- 建议等待约 `5s` 后，再查询“获取导航中的感知软件状态信息”确认定位状态

### 7.4 下发单点导航任务

**接口定义**

| 项目      | 值        |
| ------- | -------- |
| Type    | `1003`   |
| Command | `1`      |
| 消息名称    | 下发单点导航任务 |

**接口说明**

- 机器人收到该请求后，会先执行导航任务
- 任务结束后（成功 / 失败 / 取消），再返回执行结果响应

**请求参数**

| 参数                       | 含义          | 类型    | 取值 / 说明                 |
| ------------------------ | ----------- | ----- | ----------------------- |
| `Value`                  | 目标点编号       | int   | 默认可使用 `0`               |
| `MapID`                  | 目标点所在栅格地图编号 | int   | 默认可使用 `0`               |
| `PosX` / `PosY` / `PosZ` | 目标点坐标       | float | 地图坐标系，单位 `m`            |
| `AngleYaw`               | 目标点朝向       | float | `rad`                   |
| `PointInfo`              | 目标点类型       | int   | `0` 过渡点；`1` 任务点；`3` 充电点 |
| `Gait`                   | 导航步态        | int   | `0x3002` 平地；`0x3003` 楼梯 |
| `Speed`                  | 导航速度        | int   | `0` 正常；`1` 低速；`2` 高速    |
| `Manner`                 | 运动方式        | int   | `0` 前进；`1` 倒退           |
| `ObsMode`                | 导航停避障功能     | int   | `0` 开启；`1` 关闭           |
| `NavMode`                | 导航方式        | int   | `0` 直线导航；`1` 自主导航       |

**参数说明**

- `PointInfo`
  - 过渡点：用于克服地形和约束路径，到点精度较低
  - 任务点：到点精度较高，到达后保持站立，便于执行巡检任务
  - 充电点：到点后进入充电流程
- `Gait`
  - 平地步态：适用于平地、草地和不大于 `30°` 斜坡
  - 楼梯步态：适用于斜坡和高度不大于 `25cm` 的楼梯
- `ObsMode`
  - 无论开启与否，地形图自身安全停障仍始终有效
  - 对于直线导航，开启后表现为停障
  - 对于自主导航，开启后表现为避障
- `NavMode`
  - 直线导航失败时会直接返回规划失败
  - 自主导航应根据地形复杂度设置足够密集的路径点
  - 实际工程中建议优先使用多段直线导航替代长距离自主导航

## 8. 通用协议接口调用状态响应

### 8.1 接口说明

该响应消息集与请求消息集一一对应。任意协议请求下发后，系统都会自动回送一条通用调用状态响应，用于描述协议层调用是否成功。

### 8.2 响应规则

| 项目      | 规则        |
| ------- | --------- |
| Type    | 与对应请求保持一致 |
| Command | 与对应请求保持一致 |
| 返回时机    | 请求处理后自动返回 |

### 8.3 JSON 响应示例

```json
{
  "PatrolDevice": {
    "Type": 1007,
    "Command": 1,
    "Time": "2023-01-01 00:00:00",
    "Items": {
      "ErrorCode": 0,
      "ErrorMessage": "Success"
    }
  }
}
```

### 8.4 响应字段

| 字段             | 含义       |
| -------------- | -------- |
| `ErrorCode`    | 协议调用状态码  |
| `ErrorMessage` | 协议调用状态说明 |

### 8.5 通用错误码表

| ErrorCode | ErrorMessage                | 含义       | 说明                                           |
| --------- | --------------------------- | -------- | -------------------------------------------- |
| `0x0000`  | `Success`                   | 成功       | -                                            |
| `0xE001`  | `Unsupported data format`   | 数据格式不支持  | 非 XML / JSON                                 |
| `0xE002`  | `Data parsing failed`       | 数据解析失败   | 缺少 `PatrolDevice`、`Items`、`Type`、`Command` 等 |
| `0xE003`  | `Unsupported protocol type` | 不支持的协议   | 找不到对应 `Type` / `Command`                     |
| `0xE004`  | `Missing necessary fields`  | 缺少必要字段   | 协议字段不完整                                      |
| `0xE005`  | `Mismatched data type`      | 字段类型不匹配  | -                                            |
| `0xE006`  | `Mismatched request client` | 请求客户端不匹配 | 例如轴指令要求 2 秒内来自同一客户端                          |
| `0xE007`  | `No operation permission`   | 无操作权限    | -                                            |
| `0xE008`  | `Not allowed operation`     | 不允许的操作   | -                                            |
| `0xE009`  | `Operation failure`         | 操作失败     | -                                            |
| `0xE00A`  | `Unsupported function`      | 不支持的功能   | 例如导航功能仅 Pro 版本支持                             |
| `0xE00B`  | `Internal error`            | 内部错误     | 例如内部文件打开失败、内部文件解析失败                          |

## 9. 对接建议

为保证联调稳定，建议在实际开发中遵循以下顺序：

1. 先完成底层通信连通性验证（TCP / UDP、端口可达）
2. 固定使用 `JSON` 作为 ASDU 格式
3. 建立心跳机制，确认状态上报通路正常
4. 再逐步联调控制类接口
5. 最后联调导航 / 巡检类接口

### 9.1 推荐联调顺序

- 心跳指令
- 基础状态上报 / 异常状态上报
- 使用模式切换
- 运动状态转换 / 步态切换 / 轴指令
- 休眠、灯光、自主充电
- 定位初始化
- 导航任务下发 / 状态查询 / 取消导航

### 9.2 工程注意事项

- 导航相关接口仅 `M20 Pro` 支持
- 导航任务前需确认机器人处于“导航模式”
- 心跳停止后，状态上报可能中断
- 自主充电和清除充电状态属于高风险操作，应在明确流程下使用
- 初始化定位返回失败不一定代表实际重定位失败，需配合后续状态查询确认
